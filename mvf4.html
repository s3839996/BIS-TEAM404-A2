MVF4 - Payment

Description

The payment section will consist of a quick and easy solution to calculate ticket fares. The app will utilize the internet time and location to determine what the time is for the user's current location, which is then used to determine the fare for the user (whether its peak time or not). If the user is calculating a fare for a future booking, they can enter the location and time manually to get the corresponding fees/prices.

The app will consist of several cashless payment solutions. This includes an online terminal credit card processor, PayPal and device specific services such as (Google pay, Apple Pay, Samsung pay). In some cases, paying cash at the payment window/terminal will be available depending on the location. At the end of transaction, the user will be reminded that they can save their payment details for any future transactions. This would come with the ability to print a receipt at the end of a certain transaction, which can then be saved to the users native file system or sent as an email.


Lucid Textual Information

  • Upon tapping on 'Manage Payments,' the user will be menu with their payment methods displayed, in which they may choose to remove a payment method if needed. If this is their first time entering payment information, there will only be given a button that says 'Add Payment Method,' which will also be present if an existing user chooses to add more payment methods.

  • A variety of payment options will be supported, such as credit card, PayPal, etc. These will be select-able when booking a parking space, and can be saved within the application.

  • Each payment method will be represented by an icon of sorts, either of the card being used or an icon of the service being used (e.g, PayPal).


User Stories

"As a new-tech user, I would like to have a quick and easy payment method".

Quick Park's fast payment system will allow users to enter the payment details and make secured bookings just within a few taps. The user is only required to add the card details once and it will be stored on file so it is convenient and the user does not need to enter the same information multiple times. The card details which are placed on file can then later be edited in the case that card details have changed. The application will also prompt first-time users on how to set up the payment option to have an easy experience when using the application.


"As an individual who is tight on budget, I would like to see the transactions that have been made through the application."

'Quick Park' will indicate to the user a list of the times and the parking locations that the users have booked in the past and will display a summary of the savings and total monthly cost of the services that has been used. This way the user can gain insights about their spending's and at the same time plan their budgets properly and save money on parking by booking ahead and securing a bargain price.

Validation Criteria

There are several validation types that can be used to check data and information that is being entered by a user.

  • Format check: this is used to check if data is in the correct format. In relation to the payment page, this can be used when entering the card number, expiry date, and CVV, which all have their respective required formats. The expiry date can also be easily validating by utilising a date picker on the device.

  • Existence check: this is used to check if data has been entered at all. In relation to the payment page, all fields will need to be validated as having data entered in the field.

  • Lookup table: this is used to check for acceptable values in a table. In relation to the payment page, when the user enters an expiry date, it will need to be a date greater than that current day, rather than a past date.

  • Length check: this is used to check if data is of a particular length. In relation to the payment page, card number and CVV will need to be of their respective required lengths.

  • Type check: this is used to check the data type of the information entered. In relation to the payment page, a type check should be performed on the card number and CVV, as these can be numbers only.

We also need to determine a list of acceptance criteria to help us set a user's expectation for our application. This can be done by generating possible scenarios that a user may experience and outline the expectations of that scenario. We can generate this using the Given/When/Then template.
